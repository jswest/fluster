# fluster v1.1 Specification

## SQLite FTS5 Upgrade + Search Improvements

---

# Scope

v1.1 upgrades text filtering from client-side substring matching to:

* SQLite FTS5 full-text search
* Ranked results
* Token-aware queries
* Server-side filtering
* Scalable to 10k+ items

No UI redesign. No pipeline changes.

---

# Design Principles

* Keep search fast and deterministic
* Avoid client-side full corpus loading
* Maintain read-only client policy
* Preserve v1 architecture
* No unnecessary abstraction layers

---

# Phase 1 — Backend Schema Update (FTS Virtual Table)

## Goal

Add FTS5 virtual table over `embedding_text`.

## Requirements

Create FTS virtual table:

```sql
CREATE VIRTUAL TABLE items_fts USING fts5(
  item_id UNINDEXED,
  record_name,
  embedding_text,
  tokenize = 'unicode61'
);
```

### Notes

* `item_id` stored but not indexed
* `embedding_text` is primary searchable field
* `record_name` included for title-level search
* Use `unicode61` tokenizer
* Do not use porter stemmer in v1.1

## Acceptance

* `items_fts` exists in SQLite DB
* Queryable via `MATCH`

---

# Phase 2 — FTS Population

## Goal

Ensure FTS table is populated when items are embedded.

## Behavior

After `materialize_items` or `embed_items`:

* Insert into `items_fts`:

  * item_id
  * record_name
  * embedding_text

If re-running embedding:

* Clear FTS entries for affected items
* Reinsert

No triggers required for v1.1 (explicit inserts acceptable).

## Acceptance

* FTS table row count equals item count
* Re-embedding does not duplicate entries

---

# Phase 3 — Server Query Layer Upgrade

## Goal

Replace substring filtering with FTS search in query layer.

## File

```text
client/src/lib/server/queries/search.ts
```

## Query Shape

```sql
SELECT item_id, rank
FROM items_fts
WHERE items_fts MATCH ?
ORDER BY rank
LIMIT ?
```

### Requirements

* Use parameterized MATCH query
* Default limit: 2000
* Return:

  * item_id
  * optional rank
* Join to:

  * reduction coordinates
  * cluster assignment

## Acceptance

* Search returns correct item_ids
* Ranked ordering works
* No LIKE queries remain

---

# Phase 4 — Client Integration

## Goal

Replace client-side filtering logic with server-side FTS query.

## Changes

In plot route load function:

* If search query present:

  * Call FTS search
  * Fetch only matching items + coords
* If no search:

  * Load full dataset

## UI Behavior

* Remove client-side substring matching
* Preserve:

  * debounce (150–250ms)
  * match count display
* If result set > N:

  * display “Showing first N results”

## Acceptance

* Searching large corpus is instantaneous
* Memory usage reduced
* No full-text payload shipped to client unnecessarily

---

# Phase 5 — Token Handling + Query Parsing

## Goal

Support multi-word search safely.

## Requirements

* Tokenize user query by whitespace
* Construct MATCH query as:

```
term1 AND term2 AND term3
```

* Escape special FTS characters
* Do not allow raw FTS syntax injection

Optional (v1.1+):

* Support quoted phrases

## Acceptance

* Searching “artificial intelligence” works
* Searching multiple words requires both tokens
* No SQL injection risk

---

# Phase 6 — Ranking and Visual Emphasis

## Goal

Use FTS ranking to improve UX.

## Behavior

* Items returned ordered by rank
* Optionally:

  * Highlight top 20 results visually
  * Fade lower-ranked results

No UI redesign required.

## Acceptance

* Higher relevance items appear first
* Ranking stable across identical queries

---

# Phase 7 — Performance Safeguards

## Goal

Prevent pathological cases.

## Requirements

* Minimum query length: 2 characters
* Maximum returned items: configurable (default 2000)
* If 0 results:

  * Clear UI message
* If extremely broad query:

  * Show capped result warning

## Acceptance

* UI never freezes
* DB query times remain sub-100ms on 10k items

# v1.1 Definition of Done

You can:

* Search 10k+ items instantly
* Use multi-word queries
* Get ranked results
* Filter plot via server-side FTS
* Avoid shipping full embedding_text corpus to client
* Maintain clean query layer abstraction

---

# Non-Goals

* No fuzzy matching
* No synonym expansion
* No semantic search (vector similarity)
* No highlighting snippets
* No stemming

---

# Future Considerations (v1.2+)

* Porter tokenizer
* Phrase search with quotes
* Highlighted result snippets
* Hybrid vector + FTS search
* Search across metadata fields

---

# Final Constraint

FTS must remain:

* Deterministic
* Fast
* Transparent
* SQLite-native
* Fully local

No external search engine.

## 3. Fix Orphaned Job Cancellation

**Problem:** `fluster cancel` only sets `cancel_requested_at`. If the process was Ctrl+C'd, the job is stuck as `running` forever and blocks new runs.

**Plan:**

Modify `fluster/cli.py` `cancel` command:
1. After calling `request_cancel(conn, job_id)`, check if the process is actually still alive
2. Since we can't reliably detect if the pipeline process is running, take a simpler approach: add a `--force` flag to `cancel` that calls `mark_canceled(conn, job_id)` directly, transitioning the job to `canceled` status immediately
3. Without `--force`, keep current cooperative behavior (set flag, let pipeline check it)
4. Print a hint when cooperative cancel is used: `"If the pipeline process is no longer running, use --force."`

Also modify the `run` command:
5. When `get_active_job(conn)` finds a stuck job, check if `cancel_requested_at` is set. If so, auto-force-cancel it and proceed rather than blocking.

**Files:**
- `fluster/cli.py` — modify `cancel` (add `--force`) and `run` (auto-recover orphaned jobs)
- Reuse: `mark_canceled()` from `fluster/jobs/manager.py`

**Verify:** Ctrl+C during `fluster run` → `fluster cancel <id> --force` → `fluster run` starts a new job. Also: Ctrl+C → `fluster run` auto-recovers if cancel was already requested.

## 4. Suppress HuggingFace Warnings

**Problem:** First-run embedding spams noisy warnings: "unauthenticated requests to HF Hub", MLX weight loading reports, UNEXPECTED key notices.

**Plan:**

In `fluster/pipeline/embed.py`, before importing/loading sentence-transformers:
1. Set `os.environ["TOKENIZERS_PARALLELISM"] = "false"` (suppresses tokenizer warnings)
2. Set `os.environ["HF_HUB_DISABLE_PROGRESS_BARS"] = "1"`
3. Use `warnings.filterwarnings("ignore", message=".*unauthenticated.*")`
4. Set logging level for `sentence_transformers` and `huggingface_hub` loggers to WARNING or higher:
   ```python
   import logging
   logging.getLogger("sentence_transformers").setLevel(logging.WARNING)
   logging.getLogger("huggingface_hub").setLevel(logging.WARNING)
   ```
5. Do this at the top of `embed_items()` before `SentenceTransformer()` is instantiated

**Files:**
- `fluster/pipeline/embed.py` — add warning suppression before model load

**Verify:** `fluster run` → embedding step shows no HF warnings, just fluster's own progress logs.

## 5. Pipeline Progress Messages

**Problem:** `fluster run` is mostly silent between steps. On long runs, users can't tell what's happening.

**Plan:**

The pipeline already calls `log_job()` and `update_progress()` in `fluster/pipeline/run.py`, but these only write to the DB — nothing prints to the terminal.

Modify `fluster/cli.py` `run` command to print progress:
1. Before calling `run_pipeline()`, print the step count and what's about to happen
2. The simplest approach: add `console.print()` calls in the `run` CLI command wrapping each phase. But since `run_pipeline()` is a single call, we need a callback.
3. Add an optional `on_step` callback parameter to `run_pipeline()`:
   ```python
   def run_pipeline(conn, project_path, plan, job_id, on_step=None):
   ```
4. Call `on_step(step_name, completed, total)` after each step completes
5. In `cli.py`, pass a callback that uses `console.print()` or a Rich `Status` spinner:
   ```python
   def on_step(name, completed, total):
       console.print(f"  [{completed}/{total}] {name} done")
   ```

**Files:**
- `fluster/pipeline/run.py` — add `on_step` callback parameter, call it after each step
- `fluster/cli.py` — pass a progress callback to `run_pipeline()`

**Verify:** `fluster run` → prints step-by-step progress like:
```
[1/7] materialize done
[2/7] embed done (980 items)
[3/7] reduce done
...
```

## 7. `fluster use` / `fluster list` + Remove Project Name Arguments

**Problem:** Every command requires passing `project_name`. Tedious when you're working on one project.

**Plan:**

This is the largest change. Implement in order:

### Step 1: Active project file
- Store active project name in `~/.fluster/active_project` (plain text, just the name)
- Add to `fluster/config/settings.py`:
  ```python
  ACTIVE_PROJECT_FILE = FLUSTER_HOME / "active_project"
  ```
- Add helpers to `fluster/config/project.py`:
  ```python
  def set_active_project(project_name: str) -> None:
      """Write the active project name to disk."""

  def get_active_project() -> str | None:
      """Read the active project name, or None if not set."""
  ```

- `fluster use <project_name>` — validates project exists, writes to `active_project` file, prints confirmation

- Replace `_open_project(project_name)` with `_open_project()` — reads active project from file, errors with `"No active project. Run 'fluster use <name>' first."` if not set

- `job` and `cancel` keep their `job_id` argument; `export` keeps `--cluster-run`; `ingest-rows` keeps `csv_path`

### Step 4: `fluster init` auto-uses
- After `create_project()`, call `set_active_project(project_name)`
- Print: `"Created and now using project '{project_name}'"`

### Step 5: Update tests

**Files:**

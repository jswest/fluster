# fluster v1 Client Specification

## Scope

* SvelteKit v5 client in `client/`
* TypeScript
* No external UI component libraries
* Optional: `lucide-svelte` for icons
* Read-only access to `project.db`
* Relational tables only (no vec tables)
* Database path provided via `FLUSTER_DB_PATH`
* Project name provided via `FLUSTER_PROJECT_NAME`
* Launched via `fluster show` / `fluster chill` (CLI commands owned by backend team)
* Client-side filtering (FTS deferred to v1.1)
* Font: Iosevka Term (monospaced, geometric, condensed)
* No tests in v1 scope

---

# Phase 1 — Minimal Design System

## Goals

* Stripped-down styling
* All styling driven by existing CSS variables
* Reusable primitives only
* No UI framework
* Switch to `adapter-node` (required for server-side SQLite access via better-sqlite3)

## Files

```
client/src/app.css
client/src/routes/+layout.svelte
client/src/lib/components/
  Button.svelte
  Pill.svelte
  Input.svelte
  Card.svelte
  EmptyState.svelte
  Spinner.svelte
```

## Requirements

### Adapter

* Replace `@sveltejs/adapter-auto` with `@sveltejs/adapter-node`
* Update `svelte.config.js` accordingly

### app.css

* Preserve existing CSS variables (Iosevka Term, color tokens)
* Define no new color tokens unless necessary
* Base element styles:

  * `html, body`
  * headings
  * `button`, `input`
* Provide minimal utilities:

  * `.container`
  * `.stack`
  * `.row`
  * `.muted`

### Pill Component

Props:

* `variant: 'neutral' | 'info' | 'success' | 'warn' | 'danger'`
* `size: 'sm' | 'md'`
* optional `icon`
* optional `title`

Used for:

* Job status
* Cluster metadata
* Tags

### Spinner component

* Should be very geometric and rectangular.

### Overall styling notes.

* Iosevka Term is condensed, monospaced, and geometric. The design should match that. No rounded corners, etc.

### Acceptance

* Layout renders Card + Pill + Button
* No third-party UI library used
* All colors derive from CSS variables
* `adapter-node` is configured

# Phase 6 — Query Layer (No API Layer)

## Goals

* Centralized server-side queries
* No `/api/*` abstraction layer

## Files

```
client/src/lib/server/queries/
  jobs.ts
  runs.ts
  items.ts
```

## Rules

* Pages import query functions
* No raw SQL in page files
* Read-only only

## Query: Scatter Plot Data (owned by `items.ts`)

The scatter plot payload requires a multi-table join:

* `items` → `rows` (for `row_name`, aliased as `record_name`)
* `items` → `representations` (for `embedding_text`, truncated as preview)
* `reduction_coordinates` (for x, y from UMAP-2D)
* `cluster_assignments` (for `cluster_id`)

This join is owned by the query layer, not the page.

## Acceptance

* All DB access goes through query modules

# Phase 9 — Canvas Scatter Plot (UMAP2)

## Route

```
/runs/[cluster_run_id]
```

(or nested `/plot`)

## Data Payload

For each item (assembled by query layer join in Phase 6):

* item_id
* x, y (from `reduction_coordinates` for UMAP-2D reduction)
* cluster_id (from `cluster_assignments`)
* record_name (from `rows.row_name`)
* embedding_text_preview (truncated from `representations.text`)

## Rendering

* HTML Canvas
* Pan + zoom
* Hover tooltip (nearest point)
* Click selects point
* Noise cluster rendered gray

## Colors

* Deterministic ordinal scale
* Sorted cluster IDs
* D3 `scaleOrdinal`
* Noise (`-1`) always neutral gray

## Filtering (Client-Side Only)

### Search Input

* Case-insensitive substring match
* Matches:

  * `record_name`
  * `embedding_text_preview`
* Tokenized AND matching
* Debounced (150–250ms)

### UI

* Show match count
* Toggle:

  * hide non-matches
  * fade non-matches
* Optional:

  * filter within selected cluster only

## Acceptance

* 2,000+ points render smoothly
* Filtering updates under ~100ms
* No server-side search required

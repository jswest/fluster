# fluster v1 Client Specification

## Scope

* SvelteKit v5 client in `client/`
* TypeScript
* No external UI component libraries
* Optional: `lucide-svelte` for icons
* Read-only access to `project.db`
* Relational tables only (no vec tables)
* Database path provided via `FLUSTER_DB_PATH`
* Launched via `fluster show` / `fluster chill`
* Client-side filtering (FTS deferred to v1.1)

---

# Phase 1 — Minimal Design System

## Goals

* Stripped-down styling
* All styling driven by existing CSS variables
* Reusable primitives only
* No UI framework

## Files

```
client/src/app.css
client/src/routes/+layout.svelte
client/src/lib/components/
  Button.svelte
  Pill.svelte
  Input.svelte
  Card.svelte
  EmptyState.svelte
  Spinner.svelte
```

## Requirements

### app.css

* Define no new color tokens unless necessary
* Use existing CSS variables
* Base element styles:

  * `html, body`
  * headings
  * `button`, `input`
* Provide minimal utilities:

  * `.container`
  * `.stack`
  * `.row`
  * `.muted`

### Pill Component

Props:

* `variant: 'neutral' | 'info' | 'success' | 'warn' | 'danger'`
* `size: 'sm' | 'md'`
* optional `icon`
* optional `title`

Used for:

* Job status
* Cluster metadata
* Tags

### Spinner component

* Should be very geometric and rectangular.

### Overall styling notes.

* The font we're using is quite condensed. It's monospaced and geomtric. The design should match that. No rounded cornders, etc.

### Acceptance

* Layout renders Card + Pill + Button
* No third-party UI library used
* All colors derive from CSS variables

# Phase 2 — Environment Contract (Fail Fast)

## Goals

* Client refuses to run without `FLUSTER_DB_PATH`

## Files

```
client/src/lib/server/env.ts
```

## Behavior

* Read `FLUSTER_DB_PATH` from `$env/dynamic/private`
* Throw clear error if missing:

  * "FLUSTER_DB_PATH not set. Use `fluster show` or `fluster chill`."

## Acceptance

* Dev server fails immediately without env var

# Phase 7 — Jobs UI

## Routes

```
/jobs
/jobs/[job_id]
```

## Features

### Jobs List

* Show recent jobs
* Columns:

  * type
  * status
  * started_at
  * finished_at
* Status rendered via `<Pill>`

### Job Detail

* Show metadata
* Show logs (last N lines)

* Stop polling when complete

* `<Pill>`
* `<EmptyState>`
* `<Spinner>`

## Acceptance

* Polling stops automatically when done

# Phase 9 — Canvas Scatter Plot (UMAP2)

## Route

```
/runs/[cluster_run_id]
```

(or nested `/plot`)

## Data Payload

For each item:

* item_id
* x
* y
* cluster_id
* record_name
* embedding_text_preview (truncated)

## Rendering

* HTML Canvas
* Pan + zoom
* Hover tooltip (nearest point)
* Click selects point
* Noise cluster rendered gray

## Colors

* Deterministic ordinal scale
* Sorted cluster IDs
* D3 `scaleOrdinal`
* Noise (`-1`) always neutral gray

## Filtering (Client-Side Only)

### Search Input

* Case-insensitive substring match
* Matches:

  * `record_name`
  * `embedding_text_preview`
* Tokenized AND matching
* Debounced (150–250ms)

### UI

* Show match count
* Toggle:

  * hide non-matches
  * fade non-matches
* Optional:

  * filter within selected cluster only

## Acceptance

* 2,000+ points render smoothly
* Filtering updates under ~100ms
* No server-side search required
